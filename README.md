# GOSTling

> Diagrams-as-code для диаграмм, которые обязаны соответствовать ГОСТ/ЕСКД — начиная с IDEF0 и Исикавы.

Название — каламбур: GOST + "-ling". Тулза должна "выращивать" из псевдокода аккуратные, стандартные диаграммы, а не рисовать их руками в Visio.

## Откуда идея

Столкнулся с этим на дипломе: нужно нарисовать IDEF0 по ГОСТ, а вменяемого инструмента нет вообще. Mermaid, PlantUML, D2 — про IDEF0 и Исикаву не слышали. Есть один заброшенный `IDEF0-SVG` на Ruby без единого коммита за годы и без всякого ГОСТ. Всё остальное — EdrawMax, ConceptDraw, Visio, не дай бог RAMUS: ручная отрисовка, зачастую платно, и по факту полтора инструмента и оба неточные.

При этом спрос очевидно растёт — LLM отлично генерирует псевдокод структуры системы, но воткнуть это в презентабельную, стандартную диаграмму до сих пор нечем.

## Что должно получиться

- DSL/псевдокод → корректная диаграмма конкретного типа, с соблюдением стандарта
- Старт с двух типов:
  - **IDEF0** — декомпозиция родитель/потомок, ICOM-стрелки, нумерация узлов
  - **Исикава** — категории причин, вложенность
- Интерфейс для правки без пересборки с нуля
- Экспорт в SVG/PNG/PDF, пригодный для вставки в диплом или техдокументацию

## MVP-чеклист

- [ ] DSL для IDEF0 (функции, ICOM-связи, декомпозиция)
- [ ] Рендер в SVG с точным соблюдением рамок/шрифтов/нумерации по стандарту
- [ ] DSL для Исикавы
- [ ] CLI или минимальный веб-редактор с превью
- [ ] Экспорт SVG → PDF/PNG

## Пока не решено

- Какой именно ГОСТ брать за основу для IDEF0 — нужно сверить требования к оформлению точно
- DSL от руки или через YAML/JSON — что удобнее, если на входе будет LLM-генерация
- CLI для старта или сразу веб-интерфейс

## Чем это отличается от готового

Mermaid/PlantUML/D2 вообще не знают про IDEF0 и Исикаву. EdrawMax/ConceptDraw/Visio — не diagrams-as-code, ручная отрисовка, платно. IDEF0-SVG — заброшен и не про ГОСТ. По факту это единственный живой code-first инструмент с прицелом на ГОСТ-совместимый рендер, да ещё и удобный для связки с LLM на входе.
